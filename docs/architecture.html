System Architecture

The system is organized into five horizontal layers. Each layer depends only on layers below it — there are no circular dependencies between modules.

Data Flow

Data enters the system from three sources, is merged and optionally preprocessed, then used to populate charts, the SQLite database, and the pipeline view.

Fallback strategy: fetcher.js tries OpenAQ first with an 8-second AbortSignal timeout, then data.gov.in, then silently falls back to the seeded synthetic generator. The source badge in the topbar reflects which source is active. All three paths produce the same data shape — downstream code doesn't know the source.

Triggers (3 automatic behaviors)

trg_alert_on_aqi

Fires AFTER INSERT on aqi_readings. Auto-creates an alert record whenever AQI ≥ 200, classified as critical (200–299) or emergency (≥ 300).

trg_update_hourly_agg

Fires AFTER INSERT on measurements. Recalculates the matching row in hourly_aggregates using INSERT OR REPLACE so aggregates stay current without a separate job.

trg_log_quality_change

Fires AFTER UPDATE OF quality_flag on measurements. Inserts a row into quality_log recording the before/after values whenever the pipeline corrects a reading.

Data Preprocessing Pipeline

A 6-step configurable pipeline (DataPipeline class in pipeline.js) transforms raw measurements into clean, analysis-ready data. Every change is recorded in a structured audit log.

🧹
Step 1
Sentinel Normalization
−999, 9999, NaN, Inf → null. Unifies all "missing" representations.
🔄
Step 2
Unit Standardization
Auto-detects ppb / ppm / °F / K via heuristics + confidence threshold. Converts to μg/m³, mg/m³, °C.
📏
Step 3
Range Validation
Hard bounds (physically impossible) → null. Soft bounds (unusual) → quality flag.
📊
Step 4
Outlier Treatment
Ensemble voting (Z-score, Modified Z-score, IQR, rate-of-change, range). Treat: clip / interpolate / rolling median.
🔍
Step 5
Missing Imputation
Linear interp (gaps ≤ 6h), forward fill, rolling median, CPCB baseline, or seasonal mean for longer gaps.
✅
Step 6
Consistency Check
PM10 ≥ PM2.5 (physical law). AQI = f(PM2.5) via EPA breakpoints. Fix violations.

Key Design Decisions

Seeded PRNG, not Math.random()

All synthetic data uses Math.sin(seed)-based deterministic generation. The same city + timestamp always produces the same AQI, making the demo reproducible and shareable. Seed = hash(cityId + timestamp).

Seasonal realism built into simulation

Delhi/Kolkata show 1.65× AQI amplification Oct–Jan (stubble burning + temperature inversion) and 0.58× in monsoon (Jun–Aug rain washout). South Indian cities are unaffected. This matches actual CPCB seasonal patterns without needing a model.

Hygroscopic PM correction in IoT simulator

Optical PM sensors overestimate at high humidity due to water droplet scattering. The simulator applies the hygroscopic growth correction PMcorrected = PM / (1 + κ·RH/(1−RH)) with κ = 0.11, matching Petters & Kreidenweis (2007).

Ensemble outlier voting

Using ≥ 2 votes from 4 methods (Z-score, Modified Z-score, IQR, rate-of-change) reduces false positives for heavy-tailed AQI distributions. A single spike during a genuine pollution event won't be flagged by Modified Z-score but will be caught if 2+ methods agree.

Long-gap imputation fallback chain

Linear interpolation is capped at 6 consecutive missing points to avoid straight-line artifacts across a 24-hour sensor outage. Longer gaps fall back to CPCB monthly baseline, preserving seasonal shape rather than hallucinating a trend.

3NF relational schema despite in-memory DB

The SQLite schema is designed as if it were going to production PostgreSQL/TimescaleDB. Normalized reference tables (parameters, data_sources) eliminate the update anomalies that would occur if WHO guideline values were embedded in each measurement row. Migration to a real backend requires no schema changes.

SVG gauge with arc trigonometry

The semicircular AQI gauge uses standard math coordinates with SVG's y-axis flip: angleRad = π − (aqi/500)×π; needle tip at (cx + R·cos θ, cy − R·sin θ). Verified: aqi=0 → left, aqi=250 → top, aqi=500 → right.

Trigger-driven hourly aggregation

Rather than a scheduled job, trg_update_hourly_agg recalculates the hourly aggregate on every measurement INSERT using INSERT OR REPLACE. This ensures aggregates are always current without needing a background process, at the cost of slightly heavier INSERTs.
